Step-by-Step Instructions for Running the Code
Open the FitnessTracker folder, navigate to the Scripts folder and open each script. Run the first script, “generate_athlete_data.m.” This script produces the athlete’s data, the foundation for this project. Files generated from this script are located in the Athletes folder, which is inside the Data folder. Next, run the “main_analysis.m” script to compare the data sets of each athlete . Next, run the main_analysis.m script to compare the data sets of each athlete. This script processes the three CSV files generated by the Data Manager and performs all the core calculations. It categorizes each workout day into one of four heart rate zones — Resting, Light, Moderate, and Vigorous — based on each athlete's post-workout heart rate. It then calculates how many minutes each athlete spent in each zone, their average recovery rate, and summary statistics like mean heart rate, average workout duration, and training consistency. Lastly, run the script “main_visualization.m” to generate figures synthesizing the athlete’s data into readable figures. To access these figures, go back to the main folder, FitnessTracker, and open the Results folder, from there navigate to figures. 


Explanation of Data File Formats and Column Meanings
Athlete Data Files: 
Athlete data files are tables of mixed data in comma separated variable files. There are six columns: “Days,” “ExerciseType,” “PreWorkoutHR,” “PostWorkoutHR,” “Duration,” and “Intensity.” Days accounts for each day in a 21 day period. Exercise type is a string of either “Running,” “Pushups,” “Squats,” “Cycling,” or “Rest.” PreWorkoutHR is the athlete’s heart rate in beats per minute prior to exercising, while PostWorkoutHR is the heart rate after exercising. Duration is the workout length in minutes. Lastly, intensity is how rigorous the workout was. This is consistent across all three athlete csv files. The last file is a text file containing each athlete’s profile in their own lines.
analysis_results: 
The analysis process generates two primary outputs to summarize these metrics: a comprehensive workspace file and a summary statistics table. The analysis_results.mat file serves as a complete Explanation of Data File Formats and Column Meanings by functioning as a full workspace save. This format preserves all variables, tables, and categorized data—such as the calculated heart rate zones and recovery rates, to allow for further use or visualization without re-running the initial logic.
summary_statistics: 
Summary statistics is a comma separated variable file with five columns. The first Column is the “AthleteLevel,” which categorizes the athletes’ skills as beginner, intermediate, and advanced. The second column is the average heart rate in beats per minute (“AvgHeartRate”). The third column is labeled “TotalMinutes.” This column tracks the total amount of time the athlete spent exercising in minutes. Next is “Consistency,” which is the standard deviation of intensity. In this case, a lower value for consistency represents better consistency, as there is less deviation. Lastly, the final column is labeled “CompositeScore.” This column contains the ratings of each athlete based on a criteria of their average intensity, consistency, efficiency, and total workout time. This rating is a value from 1-100, where 100 is a perfect score.


Summary of key findings (3-5 bullet points)
* The Advanced athlete scored highest on the composite fitness score, driven 
  by significantly more total workout time (roughly 870 min vs ~295 min for 
  Beginner) and higher average intensity (8.5 vs 4.6)
* The Advanced athlete spent the most time in the Vigorous HR zone (>160 bpm),
  consistent with their higher intensity training profile
* Contrary to expectation, the Beginner athlete had the lowest mean recovery 
  rate, meaning their HR increased the least during exercise — this reflects 
  lower workout intensity rather than better cardiovascular fitness
* The Advanced athlete trained the most consistently (lowest std of intensity),
  showing that more experienced athletes vary their effort less day-to-day
*  All athletes correctly show 0 minutes in the Resting HR zone — rest days 
  have Duration = 0 by design, so they contribute no workout time


Special notes or assumptions made
An assumption we made was that the number of rest days were referring to a week's worth of rest. Initially, I (data manager) thought that the rest days were for the entire 21 days. For example, athlete 1 would get 1-3 rest days for the 21 day cycle. After consulting with each other and the professor, we decided that the 1-3 rest days were limited to that week. Another assumption we made was that the exercise type for each day was random.


I’ve (algorithm developer) structured our analysis around a specific set of logic rules that prioritize workout duration and uniform thresholds over individual physiological variance. I identify rest days strictly by a Duration of zero, intentionally moving past the fact that heart rate values might be identical on those days. To maintain a standardized baseline across the board, I apply fixed heart rate zones at 100, 130, and 160 bpm for every athlete regardless of age.
When calculating the final fitness scores, I weighted each of the four core metrics equally at 25% to ensure no single factor overshadows the others. To keep our consistency metrics accurate, I’ve excluded all intensity values of zero on rest days from my mean and standard deviation calculations, preventing any artificial inflation of the results. Finally, I treat recovery rate as a measure of workout intensity rather than isolated cardiovascular efficiency; this means I interpret the lower rates in beginners as a reflection of easier sessions rather than superior heart health.


Reflections
Role 1 - Data Manager Reflection: 
Organization was a key aspect of this project. Depending on the file location, certain functions, like the fullfile, had to be modified. I chose to follow the examples provided in the project 2 pdfs. The main folder contained three separate folders: Data, Scripts, and Results, with subfolders. This created an organized environment, where file locations can easily be identified. However, we also ran into some problems with committing changes to the shared files. Files that were initially created in the wrong location were moved to the correct location. Editing those files produced additional complications, as MATLab claimed the file paths did not exist. We were able to overcome this with trial and error. Overall the organization and data management were beneficial to the project, with a few hiccups along the way.


Role 2 - Algorithm Developer Reflection:


The most useful troubleshooting strategy was incremental testing, building the script section by selection and using fprint print statements to verify results at each stage before moving on. For example, after loading the CSV files I printed the table dimensions to confirm the data loaded correctly, and after zone categorization I printed zone assignments per athlete so I could catch any logic zeros early. This made it much easier to isolate where a bug came from rather than running the whole script at once and getting a confusing error at the end. 


I followed several practices to make the script readable by anyone on the team, not just me. Every script starts with a full header block listing the purpose, inputs, outputs, team members, and date. I used ‘%%’ section breaks to divide the script into logical  stages so someone could jump straight to “Task 4” without reading everything. Variable names are descriptive (meanPostHR instead x, isWorkout instead of a flag), and any “magic numbers” like the 1e-9 epsilon or the zone threshold of 100/130/160 bpm - have inline comments explaining the reasoning. I also used fullfile() error checking so the script runs on any operating system, and added isfile() error checking so the script fails with a clear message rather than a cryptic MATLAB error if a data file is missing. 


Role 3 - Visualization Specialist Reflection:
To effectively communicate our results, I utilized a consistent color-coding scheme across multiple plots to represent the three distinct athlete levels (e.g., blue for Beginner, green for Intermediate, and red for Advanced). I ensured all plots adhered to strict engineering standards by including descriptive titles, proper axis labels with units (bpm, minutes), distinct legends, and gridlines for readability. I chose a grouped bar chart for the HR Zone analysis to allow for immediate, side-by-side comparison of discrete time categories. Scatter plots were used for the Duration vs. Heart Rate data to easily identify clusters and outliers across fitness levels, while line graphs were used for the 21-day trend to best display continuous changes over time.
